# MiniRTC 竞品与技术调研报告

**调研日期**: 2026-03-09
**项目**: MiniRTC - 轻量级 WebRTC 库

---

## 一、竞品分析

### 1. 主流 WebRTC 开源项目

| 项目 | 语言 | 特点 | 适用场景 |
|------|------|------|----------|
| **Google WebRTC** | C++ | 官方实现，功能最全，庞大 | 浏览器、生产级应用 |
| **Pion WebRTC** | Go | 纯 Go 实现，现代化 API | Go 项目快速集成 |
| **Janus Gateway** | C | WebRTC 服务器，插件化 | SFU/MCU 服务器 |
| **ZLMediaKit** | C++ | 多协议支持 (RTSP/RTMP/WebRTC) | 媒体服务器 |
| **LiveKit** | Go | 云原生，可扩展 | 规模化部署 |
| **SRS** | C++ | 高性能 RTMP/WebRTC | 直播/实时通信 |

### 2. 轻量级 WebRTC 库

| 项目 | 语言 | 特点 |
|------|------|------|
| **libsrtp** | C | SRTP 加密，仅传输层 |
| **libwebrtc (封装)** | C++ | 简化 Google WebRTC |
| **WebRTC++** | C++ | 轻量替代品 (较少维护) |

## 二、技术调研

### 1. 架构分层

```
┌─────────────────────────────────────┐
│           应用层 API                │
├─────────────────────────────────────┤
│  Session Signaling               │
├──────── /──────┬──────────────────────┤
│  VoiceEngine │    VideoEngine      │
│  - Codec     │    - Codec          │
│  - AEC/NR    │    - Jitter Buffer  │
├──────────────┴──────────────────────┤
│         Transport Layer             │
│  - RTP/RTCP                        │
│  - ICE/STUN/TURN (可选)            │
└─────────────────────────────────────┘
```

### 2. 关键技术组件

#### 音频编解码
| Codec | 特点 | 场景 |
|-------|------|------|
| **Opus** | 支持 6kbps-510kbps, 低延迟 | WebRTC 默认 |
| **iLBC** | 窄带，15.2kbps | 兼容性 |
| **G.711** | 64kbps，PCM | 传统电话 |

#### 视频编解码
| Codec | 特点 | 场景 |
|-------|------|------|
| **VP8/VP9** | 开源，免专利费 | WebRTC 原生 |
| **H.264** | 兼容性好，专利复杂 | 广泛支持 |

#### 传输层
| 协议 | 说明 |
|------|------|
| **RTP** | 实时传输协议 |
| **RTCP** | 传输控制 |
| **SRTP** | RTP 加密 |
| **ICE/STUN/TURN** | NAT 穿透 (暂不需要) |

### 3. 依赖库选择

| 组件 | 推荐库 | 备选 |
|------|--------|------|
| 音频编码 | libopus | iLBC (内置) |
| 视频编码 | libvpx (VP8/VP9) | x264 (H.264) |
| RTP传输 | 自实现 / libsrtp | libjingle |
| 网络库 | libuv / Boost.Asio | 原生 socket |
| 构建系统 | CMake | Meson |

## 三、技术路线建议

### 方案 A: 自底向上 (推荐)
1. 先实现 RTP 传输层
2. 集成 Opus 音频编解码
3. 集成 libvpx 视频编解码
4. 实现 Fake 音视频采集/播放接口
5. 组装端到端测试

**优点**: 轻量可控，适合学习
**缺点**: 工作量较大

### 方案 B: 基于现有库
1. 使用 libwebrtc 封装
2. 仅做 API 简化层

**优点**: 功能完整
**缺点**: 依赖重，不符合"轻量"定位
